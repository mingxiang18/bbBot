# 每日资讯日报功能 — 需求文档

> 状态：草案（v0.1）｜最后更新：2026-05-30
> 本文档基于一次完整的方案调研 + 真实数据 demo + 与需求方的四项关键决策确认编写。
> 文中所有数据源均为 **2026-05-30 当天实测**（拉取真实 RSS 验证存活与更新频率），非凭空罗列。

---

## 1. 背景与目标

为 bbBot 新增一个「每日资讯日报」功能：每天定时从一批**权威新闻源**采集资讯，经 **AI 整理**（去重、摘要、分类、排序）后，生成一个**可交互的 HTML 网页日报**并托管，供查看。

**核心诉求（来自需求方原话）**：

- 收集每日新闻，整理后以 **HTML 形式**展现。
- 形态是**可点击、可交互的 HTML 网页**，不是图片。
- 数据源要**权威、稳定、好用，不要小道消息/热搜八卦向**。

---

## 2. 本期范围

### 2.1 本期要做（In Scope）

1. 多源 RSS 采集（官方直连源 + 自建 RSSHub 源）。
2. 采集结果去重 + 轻量持久化。
3. 接入项目现有 LLM 做整理：去重聚类、生成中文摘要、分类、重要性排序、生成「今日速览」。
4. 生成自包含的**交互式 HTML 日报**（分类筛选、折叠、原文跳转、响应式、深色模式）。
5. 将 HTML 托管到项目静态目录 `/bot/static`，对外给出可访问 URL。
6. 每日定时自动执行上述流程。

### 2.2 本期不做（Out of Scope，已确认）

- **不做自动推送**到聊天平台（Telegram/QQ/Discord）。首版只生成页面并给出 URL，验证内容质量后再议。但**架构上为推送预留接口**（见 §4.7）。
- 不做用户个性化订阅 / 多用户偏好。
- 不做评论、点赞等社交互动。
- 不做全文抓取与全文存储（仅用 RSS 提供的标题/摘要/链接；正文以「阅读原文」跳转到源站）。

## 4. 功能需求

### 4.1 数据采集（NewsFetcher）

- **FR-1.1** 支持从配置化的源列表采集，每个源含：名称、分类、URL、获取方式（官方直连 / RSSHub）、语言（中/英）。
- **FR-1.2** 用项目现有 `RestClient` 拉取，用 jsoup/标准 XML 解析 RSS 2.0，提取每条 `title / link / description / pubDate`。
- **FR-1.3** **单源失败隔离**：任一源超时/解析失败/被反爬，记录日志并跳过，不影响其他源与整体流程。
- **FR-1.4** 每源设置抓取条数上限（默认 30 条/源，可配），避免长尾源灌入。
- **FR-1.5** 对英文源，标题原样保留（D7）。

### 4.2 去重与存储（NewsStore）

- **FR-2.1** 入库前按 `link`（归一化后）+ 标题哈希做**跨天去重**，避免同一新闻隔天重复出现。
- **FR-2.2** 持久化采用 MyBatis-Plus + MySQL（复用现有 ORM），单表存储当日采集到的条目及其 AI 整理结果。
- **FR-2.3** 保留历史归档若干天（默认 30 天，可配），支持回看过去某天的日报页面。
  - *待确认项 TBD-1：归档保留天数默认 30，是否调整见 §10。*

### 4.3 AI 整理（NewsAiCurator）

复用项目现有 LLM Provider（`OpenAiCompatProvider` / `AnthropicProvider`）。每日**一次**批处理，输入为去重后的当日条目（仅标题+原始摘要，不喂全文以控成本）。

- **FR-3.1 去重聚类**：识别同一事件的多源报道，合并为一条，标注「多源合并 / 去重 N→1」。
- **FR-3.2 中文摘要**：为每条生成一句话中文摘要（含英文源，D7）。
- **FR-3.3 分类**：将每条归入固定分类集合（见 §4.4 分类定义）。
- **FR-3.4 重要性排序**：为每条打重要性分（用于排序与 ★ 标记）。
- **FR-3.5 今日速览**：生成全局摘要导语（按板块归纳当日要点）。
- **FR-3.6 成本控制**：喂给 LLM 的总条数设上限（默认 ≤ 40 条，可配）；一天一次调用。
- **FR-3.7 降级**：LLM 调用失败时，降级为「不整理」直接用原始标题+原始摘要出页（保证日报仍能产出）。
  - *待确认项 TBD-2：AI 整理的子项范围以上述 6 项为默认，是否增减见 §10。*

### 4.4 页面生成（NewsPageBuilder）

产出一个**自包含**的交互式 HTML 文件（内联 CSS + 少量原生 JS，无外部框架依赖）。

**分类集合（默认 8 类，可配）**：🤖 AI、🌏 国际、🏛 时政、💰 财经、🍎 苹果数码、🎮 游戏、🚗 汽车、🔬 科技。

- **FR-4.1** 顶部：标题 + 日期 + 「今日速览」导语块。
- **FR-4.2** 分类 Tab：点击实时筛选对应分类、显示计数、吸顶（sticky）。
- **FR-4.3** 每条卡片含：来源徽章、分类、英文源 `EN` 角标、★重要性、AI 中文摘要、「阅读原文 →」**跳转真实文章 URL**（`target="_blank"`）。
- **FR-4.4** 链接必须是**逐篇文章真实 URL**（取自 RSS `<link>`），不得用站点首页占位。
- **FR-4.5** 响应式（手机/桌面自适应）+ 跟随系统深色模式。
- **FR-4.6** 每类展示条数上限（默认 5 条/类，可配）。

> 注：demo 阶段已用 Python 原型 `gen.py` 跑通「真实 XML → 解析 → 去重 → 分类 → 出页」，本期将该逻辑平移为 Java 的 `NewsPageBuilder`。

### 4.5 托管与访问（D6）

- **FR-5.1** 生成的 HTML 写入项目静态目录 `/bot/static`（运行时挂 NFS PVC `bb-bot-static-pvc`），按日期命名，如 `/bot/static/news/2026-05-30.html`。
- **FR-5.2** 提供对外访问入口（静态映射或一个 Controller），可通过 URL 直接打开当日及历史日报。
- **FR-5.3** 提供一个「最新」入口（如 `/news/latest` 或固定 `index.html`）始终指向当天。
  - *待确认项 TBD-3：对外访问路径/域名前缀，依现有静态资源暴露方式确定。*

### 4.6 调度（DailyNewsSchedule）

- **FR-6.1** 每日定时触发完整流程（采集→去重→AI 整理→生成→托管）。
- **FR-6.2** 默认触发时间 **每日 08:00**（可配 cron）。
- **FR-6.3** 实现方式默认用 Spring `@Scheduled`（参考现有 `TmpFileCleanSchedule`）；若需运行时动态调整，可改用现有 `ai_cron_task` 机制。
  - *待确认项 TBD-4：触发时间 08:00 与实现方式（@Scheduled）为默认，是否调整见 §10。*

### 4.7 推送（本期不做，预留接口）

- **FR-7.1** 首版不自动推送（D5）。
- **FR-7.2** 但在设计上把「产出日报 URL」与「投递」解耦：保留一个可选的投递步骤接口，未来接 `BbMessageApi.sendMessage()` 推送「📰 今日资讯 + 链接」到指定平台/群即可启用，无需重构主流程。

## 6. 技术方案与复用映射

整体流水线（每日一次）：

```
DailyNewsSchedule (@Scheduled, 默认每日 08:00)
  → NewsFetcher        RestClient 拉官方源 + 自建 RSSHub 源，解析 RSS，单源 try-catch 隔离
  → NewsStore          MyBatis-Plus 入库，按 link 归一化跨天去重
  → NewsAiCurator      调现有 LLM：去重聚类 + 中文摘要 + 分类 + 重要性排序 + 今日速览
  → NewsPageBuilder    生成自包含交互式 HTML（分类 Tab + 折叠 + 真实原文链接）
  → NewsHosting        写入 /bot/static/news/<date>.html + 更新 latest 入口
  → (预留) NewsPusher   本期不启用；未来接 BbMessageApi 推链接
```

**复用现有基础设施**（来自代码库调研）：

| 能力 | 复用 | 新增 |
|---|---|---|
| HTTP 采集 | `RestClient`（带代理/超时）+ jsoup | — |
| 持久化 | MyBatis-Plus `BaseMapper<T>` + MySQL | `database/news/` 模块 |
| AI 整理 | `OpenAiCompatProvider` / `AnthropicProvider` | `NewsAiCurator` |
| 页面生成 | 参考 `SplatoonHtmlRenderer` 的字符串模板思路（但输出 HTML 文件，非图片） | `NewsPageBuilder` |
| 静态托管 | `/bot/static`（NFS PVC `bb-bot-static-pvc`）、`ResourcesUtils` | 静态映射/Controller |
| 调度 | `@Scheduled`（参考 `TmpFileCleanSchedule`）或 `ai_cron_task` | `DailyNewsSchedule` |
| 推送（预留） | `BbMessageApi.sendMessage(BbSendMessage)` | `NewsPusher`（本期不实现） |

**新增代码结构（建议）**：

```
bb-bot-server/.../handler/news/
  NewsFetcher.java
  NewsAiCurator.java
  NewsPageBuilder.java
  NewsHosting.java
bb-bot-server/.../database/news/
  entity/NewsItem.java
  mapper/NewsItemMapper.java
  service/NewsItemService.java
bb-bot-server/.../schedule/
  DailyNewsSchedule.java
bb-bot-server/.../config/
  NewsConfig.java   (@ConfigurationProperties(prefix="news"))
```

## 7. 配置项设计（application.yml）

```yaml
news:
  enabled: true
  cron: "0 0 8 * * ?"          # 默认每日 08:00（TBD-4）
  archiveDays: 30              # 历史保留天数（TBD-1）
  perSourceLimit: 30          # 每源抓取上限
  perCategoryLimit: 5         # 每类展示上限
  hosting:
    dir: /bot/static/news     # 落盘目录（D6）
    publicBase: /news         # 对外访问前缀（TBD-3）
  rsshub:
    baseUrl: http://rsshub:1200   # 自建 RSSHub 内网地址（D4）
  ai:
    enabled: true
    maxItems: 40              # 喂 LLM 的总条数上限（成本控制）
    translateEnTitle: false   # 英文标题不翻译，摘要中文（D7）
  push:
    enabled: false            # 首版不自动推送（D5）
  sources:
    - { name: 澎湃新闻,   category: 时政, via: rsshub,  path: /thepaper/featured, lang: zh }
    - { name: 中新网滚动, category: 时政, via: direct,  url: https://www.chinanews.com.cn/rss/scroll-news.xml, lang: zh }
    - { name: 电动邦,     category: 汽车, via: rsshub,  path: /diandong/news, lang: zh }
    # ... 其余见 §5.1 完整清单
```

> 密钥（如 LLM api-key）沿用项目现有 `ai.models.*` 配置，不在 `news` 段重复。

## 8. 非功能需求

- **NFR-1 稳定性**：单源失败不影响整体；LLM 失败可降级出页（FR-3.7）。
- **NFR-2 权威性**：仅采用 §5.1 经实测的权威源；新增源上线前须同样实测「存活 + 更新频率」。
- **NFR-3 成本**：AI 整理每日一次、仅喂标题+摘要、条数封顶，单次成本可控（约数千 token 量级）。
- **NFR-4 性能**：采集并发拉取、单次完整流程目标在分钟级完成。
- **NFR-5 可维护**：源列表纯配置化，增删源不改代码；分类集合可配。
- **NFR-6 容错**：长图/超长列表通过每类条数上限控制；RSS 字段缺失时优雅降级（参考 Splatoon 渲染器的缺字段降级思路）。

## 9. 部署影响（重要，关联发布链路）

> ⚠️ 依 `CLAUDE.md`：push master 即自动构建并部署到生产，改动需保证 auto-deploy 跑通。

接入 RSSHub（D4）意味着集群里要**多一个 RSSHub 服务**：

- **DEP-1** 在 namespace `bb-bot` 增加一个 RSSHub Deployment（镜像 `diygod/rsshub`）+ Service，bb-bot 通过内网地址（如 `http://rsshub:1200`）访问。
- **DEP-2** 须在 `scripts/deploy/k8s/bb-bot/bb-bot.yaml` 中纳入该组件（YAML 写错会导致 `kubectl apply` 失败 → 整个 release 失败，需谨慎）。
- **DEP-3** RSSHub 走**自建实例**而非公共实例（公共实例反爬严重、有缓存延迟）；实测确认默认镜像**已移除汽车之家/懂车帝**等路由，故汽车依赖电动邦路由。
- **DEP-4** 资源占用小（Node 服务），可设较小的 resources/limits。
- **DEP-5** 新增静态访问入口若涉及端口/路由变化，同样需在 `bb-bot.yaml` 体现。

> *待确认项 TBD-5：RSSHub 是与 bb-bot 同 namespace 单独 Deployment，还是其他部署形态，部署阶段细化。*

## 11. 里程碑 / 任务拆分（建议）

1. **M1 数据层**：`database/news` 模块（entity/mapper/service）+ 去重逻辑。
2. **M2 采集**：`NewsFetcher`（官方直连 + RSSHub）+ 源配置 + 单源隔离。
3. **M3 AI 整理**：`NewsAiCurator` 接现有 LLM，含降级。
4. **M4 页面生成**：`NewsPageBuilder`（由 demo `gen.py` 逻辑平移），交互式 HTML。
5. **M5 托管 + 调度**：写 `/bot/static` + 对外入口 + `DailyNewsSchedule`。
6. **M6 部署**：RSSHub 组件入 `bb-bot.yaml`，打通 auto-deploy。
7. **M7（后续）推送**：启用 `NewsPusher`（本期不做）。
