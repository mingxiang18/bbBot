# 每日资讯日报功能 — 详细设计文档

> 状态：草案（v0.1）｜最后更新：2026-05-30
> 配套文档：[需求文档.md](./需求文档.md)（本文不重复背景，仅承接其结论做详细设计与任务拆分）
> 设计目标：把工作拆成**边界清晰、可并行独立开发、互不重合**的任务。所有任务通过 §4「共享契约」这一**唯一接缝**对接；除契约外，每个任务**独占一组互不相交的文件/包**。

## 3. 总体架构与数据流

```
                    ┌─────────────── 共享契约（T0，唯一接缝）───────────────┐
                    │  DTO: NewsItem · CuratedItem · DailyReport · ReportMeta │
                    │  接口: NewsFetcher / NewsStore / NewsAiCurator /         │
                    │        NewsPageBuilder / NewsHosting                     │
                    │  AI 输出 JSON schema · DB DDL · news 配置 schema          │
                    └─────────────────────────────────────────────────────────┘
                                          ▲ 所有模块仅依赖契约，互不直接依赖

DailyNewsSchedule (T6, @Scheduled 每日 08:00) —— 编排者，只依赖接口
   │
   ├─(1)→ NewsFetcher.fetchAll()         (T2)  ── List<NewsItem>
   ├─(2)→ NewsStore.dedupAndSave(items)  (T1)  ── List<NewsItem>（去重后的新条目）
   ├─(3)→ NewsAiCurator.curate(items)    (T3)  ── DailyReport
   ├─(4)→ NewsStore.saveReport(report)   (T1)  ── 持久化整理结果（供归档回看）
   ├─(5)→ NewsPageBuilder.buildDaily(report, availableDates)  (T4) ── HTML
   ├─(6)→ NewsPageBuilder.buildArchiveIndex(metas)            (T4) ── 归档索引 HTML
   └─(7)→ NewsHosting.publish(date, dailyHtml, indexHtml)     (T5) ── 落盘 + URL

NewsViewController (T5) ── 对外暴露 /news/**（无鉴权），读 /bot/static/news/ 下文件
```

**关键设计原则**：
- 模块之间**只通过契约对象与接口通信**，不互相 import 实现类。
- `DailyNewsSchedule`（T6）是唯一知道"全流程顺序"的地方；各功能模块对流程无感知。
- 去重的**唯一归属是 T1（NewsStore）**；AI 的聚类合并（语义级）归 T3，二者层次不同（见 §5 去重边界说明），不重合。

## 4. 共享契约（Task T0 产出 —— 所有人开工前的唯一依赖）

> T0 只产出**类型、接口、配置骨架、DDL**，不含业务实现。T0 合入后，T1–T6 可并行对着这些契约开发，缺的依赖用 stub。

### 4.1 DTO 定义

```java
package com.bb.bot.handler.news.contract;

/** 采集到的原始条目（NewsFetcher 产出 / NewsStore 入参） */
public record NewsItem(
    String sourceName,     // 源名，如 "中新网"
    String category,       // 源声明的分类（可空，最终分类以 AI 为准）
    String title,          // 原始标题（英文源保留英文）
    String link,           // 逐篇文章真实 URL（必填，不得为首页）
    String description,    // 原始摘要（可空）
    String pubDate,        // 原始发布时间字符串
    String lang,           // "zh" | "en"
    String linkHash        // 由 link 归一化后算的去重键（T2 填充）
) {}

/** AI 整理后的条目（NewsAiCurator 产出，进 DailyReport） */
public record CuratedItem(
    String title,          // 标题（英文源保留英文 —— D7）
    String link,
    String sourceName,
    String category,       // 最终分类，取值见 §4.4 分类枚举
    String summaryZh,      // 中文一句话摘要
    int importance,        // 1–5，用于排序与 ★
    boolean english,       // 是否英文源（决定 EN 角标）
    int mergedCount,       // 多源合并数量，1 表示未合并
    String note            // 角标备注，如 "多源合并" / "去重 2→1" / ""
) {}

/** 一天的完整日报（NewsAiCurator 产出 / NewsPageBuilder 入参） */
public record DailyReport(
    String date,                 // "2026-05-30"
    String brief,                // 今日速览导语
    java.util.List<CuratedItem> items,   // 已排序、已分类
    int sourceCount,             // 聚合源数
    int totalCount               // 精选条数
) {}

/** 归档元信息（NewsStore 产出 / 归档索引页用） */
public record ReportMeta(
    String date,
    int totalCount,
    int sourceCount,
    String url                   // 该日报的访问 URL
) {}
```

### 4.2 模块接口

```java
package com.bb.bot.handler.news.contract;
import java.util.List;

public interface NewsFetcher {            // T2 实现
    List<NewsItem> fetchAll();            // 拉全部已配置源，单源失败隔离
}

public interface NewsStore {              // T1 实现
    List<NewsItem> dedupAndSave(List<NewsItem> items);  // 跨天去重，返回"新"条目
    void saveReport(DailyReport report);                // 存整理结果
    DailyReport getReport(String date);                 // 回看某天
    List<ReportMeta> listRecent(int days);              // 归档列表（默认 30）
}

public interface NewsAiCurator {          // T3 实现
    DailyReport curate(List<NewsItem> items);  // 去重聚类/摘要/分类/排序/速览；失败降级
}

public interface NewsPageBuilder {        // T4 实现
    String buildDaily(DailyReport report, List<ReportMeta> availableDates); // 单日页（含交互JS）
    String buildArchiveIndex(List<ReportMeta> metas);                       // 归档索引页
}

public interface NewsHosting {            // T5 实现
    /** 落盘单日页 + 归档索引 + 更新 latest，返回单日页 URL */
    String publish(String date, String dailyHtml, String archiveIndexHtml);
}
```

### 4.3 AI 输出 JSON 契约（T3 与 T4 的对接基准，必须一致）

`NewsAiCurator` 要求 LLM 返回如下 JSON，再反序列化为 `DailyReport`：

```json
{
  "brief": "今日速览导语……",
  "items": [
    {
      "title": "……", "link": "https://……", "sourceName": "量子位",
      "category": "AI", "summaryZh": "……", "importance": 5,
      "english": false, "mergedCount": 2, "note": "去重 2→1"
    }
  ]
}
```

- `category` 取值必须 ∈ §4.4 枚举。
- `link` 必须回填为原条目真实链接（合并时取代表条目的链接）。
- LLM 不可靠时（解析失败/超时）→ T3 走降级：`brief=""`，`items` 由原始 `NewsItem` 直接映射（`summaryZh=description`、`importance=3`、`category=源声明分类或"科技"`）。

### 4.4 分类枚举（固定 8 类，配置可改顺序/增删）

```
AI, 国际, 时政, 财经, 苹果, 游戏, 汽车, 科技
```
展示标签（emoji + 名）由 T4 维护映射表；分类**键**由 T0 定义为常量，T3/T4 共用。

### 4.5 数据库 DDL（T1 拥有，T0 提供建表脚本）

```sql
-- 原始 + 整理后条目（一行一条）
CREATE TABLE news_item (
  id           BIGINT PRIMARY KEY AUTO_INCREMENT,
  report_date  DATE NOT NULL,
  source_name  VARCHAR(64) NOT NULL,
  category     VARCHAR(16),
  title        VARCHAR(512) NOT NULL,
  link         VARCHAR(1024) NOT NULL,
  link_hash    CHAR(40) NOT NULL,
  description   TEXT,
  pub_date     VARCHAR(64),
  lang         VARCHAR(8),
  summary_zh   VARCHAR(512),
  importance   TINYINT,
  cluster_key  VARCHAR(64),
  merged_count INT DEFAULT 1,
  created_at   DATETIME DEFAULT CURRENT_TIMESTAMP,
  UNIQUE KEY uk_link_hash (link_hash),     -- 跨天去重
  KEY idx_date (report_date)
);

-- 每日报告元信息（速览 + 统计 + 落盘路径）
CREATE TABLE news_daily (
  report_date  DATE PRIMARY KEY,
  brief        TEXT,
  total_count  INT,
  source_count INT,
  html_path    VARCHAR(256),
  generated_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

## 5. 去重的边界划分（防止 T1 与 T3 重合，单独说明）

去重分两层，**归属不同任务、互不重合**：

| 层 | 含义 | 归属 | 实现 |
|---|---|---|---|
| L1 物理去重 | 同一篇文章（同 link）跨天/跨源重复 | **T1 NewsStore** | `link_hash` 唯一键，纯字符串/哈希比对，无语义 |
| L2 语义聚类 | 不同链接但同一事件的多源报道合并 | **T3 NewsAiCurator** | 交给 LLM 聚类，产出 `mergedCount/note` |

T1 只做 L1（确定性、可单测）；T3 只做 L2（语义、靠 LLM）。T6 编排顺序保证 L1 在 L2 之前（先物理去重再喂 LLM）。

## 6. 各模块详细设计

### 6.1 T2 NewsFetcher（采集）
- **输入**：`NewsConfig.sources`。**输出**：`List<NewsItem>`。
- **逻辑**：遍历源 → 按 `via`（direct / rsshub，rsshub 拼 `rsshub.baseUrl + path`）用 `RestClient` 拉取 → 解析 RSS 2.0 取 `title/link/description/pubDate` → 填 `lang`、算 `linkHash`（link 去 query 归一化后 SHA1）→ 每源截断到 `perSourceLimit`。
- **容错**：单源 try-catch，失败记日志并跳过（FR-1.3）。
- **不负责**：去重、入库、分类（分别属 T1/T3）。
- **独占文件**：`handler/news/fetch/NewsFetcherImpl.java`、`handler/news/fetch/RssParser.java`。
- **单测**：用本地 RSS 样例 XML（fixture）验证解析正确、缺字段降级、单源失败不影响其余、linkHash 稳定。

### 6.2 T1 NewsStore（去重 + 持久化 + 归档）
- 实现 `dedupAndSave`（L1 物理去重，返回新条目）、`saveReport`、`getReport`、`listRecent`。
- 复用 MyBatis-Plus `BaseMapper`；保留天数 30，过期清理（可在此模块加一个清理方法，由 T6 调用或独立 @Scheduled —— 归 T1 拥有）。
- **独占文件**：`database/news/entity/{NewsItemPo,NewsDailyPo}.java`、`mapper/*`、`service/*`、`handler/news/store/NewsStoreImpl.java`。
- **单测**：去重命中/不命中、跨天去重、归档列表按日期倒序、保留天数裁剪。

### 6.3 T3 NewsAiCurator（AI 整理）
- 调现有 `OpenAiCompatProvider`/`AnthropicProvider`；构造 prompt（输入仅 title+description，≤ `ai.maxItems`）；要求按 §4.3 JSON 输出；反序列化为 `DailyReport`；做 L2 语义聚类、中文摘要、分类、重要性排序、生成 brief。
- **降级**（FR-3.7 / §4.3）：异常时不抛，返回由原始条目直接映射的 `DailyReport`。
- **独占文件**：`handler/news/curate/NewsAiCuratorImpl.java`、`handler/news/curate/CuratePrompt.java`、`handler/news/curate/CurateResponse.java`（JSON 映射）。
- **单测**：给定 mock LLM JSON → 正确产出 DailyReport；LLM 抛异常 → 走降级且不抛；分类越界值被纠正到合法枚举。

### 6.4 T4 NewsPageBuilder（页面生成 + 全部交互）
- 把 demo 的 `gen.py` 逻辑平移为 Java 模板（字符串模板，内联 CSS + 原生 JS，无框架）。
- **交互（本任务全包）**：
  - 分类 Tab 实时筛选 + 计数 + sticky（demo 级）。
  - 卡片折叠展开。
  - **站内搜索框**：纯前端 JS，按关键词过滤当前页卡片（标题+摘要）。
  - **跨天往期导航**：单日页顶部放日期切换器（上一天/下一天/选日期，数据来自入参 `availableDates`）；另产出**归档索引页** `buildArchiveIndex`。
- 英文源：标题英文 + `EN` 角标，摘要中文（D7）。
- **独占文件**：`handler/news/render/NewsPageBuilderImpl.java`、`render/NewsHtmlTemplate.java`、`render/assets/`（CSS/JS 片段）。
- **单测**：golden 快照测试（参考仓库 golden 测试习惯）——给定固定 `DailyReport` fixture → 生成 HTML 与 golden 文件比对；断言所有 `href` 为 http 真链接、分类计数正确、搜索/导航的 DOM 钩子存在。

### 6.5 T5 NewsHosting（托管 + 对外访问）
- `publish`：把单日 HTML 写 `news.hosting.dir`（`/bot/static/news/<date>.html`）、写归档索引 `index.html`、更新 `latest.html`（或重定向），返回 URL（`publicBase + /<date>.html`）。
- `NewsViewController`：暴露 `/news/**`（无鉴权），返回静态文件；`/news/` → 最新，`/news/<date>.html` → 指定日，`/news/archive` → 索引。
- **独占文件**：`handler/news/hosting/NewsHostingImpl.java`、`controller/NewsViewController.java`。
- **单测**：路径拼装、latest 更新、日期非法输入处理；Controller 用 MockMvc 验证可访问、无鉴权、404 行为。

### 6.6 T6 DailyNewsSchedule（编排）
- `@Scheduled(cron = "${news.cron}")` 串起 (1)–(7)（见 §3）；只依赖**接口**，构造期注入；可对缺失实现用 stub 跑通编排单测。
- 总开关 `news.enabled`；全流程包一层异常保护，任一步失败记日志，不影响下次触发。
- **独占文件**：`schedule/DailyNewsSchedule.java`。
- **单测**：用 mock 的五个接口验证调用顺序与参数传递、enabled=false 时不执行、某步抛异常时被吞并记录。

### 6.7 T7 部署（RSSHub + 发布链路）
- 在 `scripts/deploy/k8s/bb-bot/bb-bot.yaml` 增加 RSSHub Deployment（`diygod/rsshub`）+ Service（内网 `http://rsshub:1200`），小 resources。
- 确认 `news.rsshub.baseUrl` 指向该 Service；确认 auto-deploy（push master）链路不被破坏（YAML 合法、apply 通过）。
- **独占文件**：`scripts/deploy/k8s/bb-bot/bb-bot.yaml`（仅追加 RSSHub 段）、必要的 `application-prod.yml` 的 `news` 覆盖。
- **验收**：本地 `kubectl apply --dry-run` 通过；RSSHub 镜像采用源路由实测可用（已在调研阶段验证）。

## 7. 任务拆分总表（边界 / 依赖 / 不重合声明）

> 依赖只有一条硬边：**T0 必须先合入**（它是契约）。之后 T1–T6 **可完全并行**；T7 与代码任务并行。

| 任务 | 标题 | 独占文件/包（互不相交） | 依赖 | 对外提供 | 消费 | 完成标准 |
|---|---|---|---|---|---|---|
| **T0** | 契约与骨架 | `handler/news/contract/*`、`config/NewsConfig.java`、`application.yml` 的 `news` 段、建表 SQL | 无 | 全部 DTO/接口/DDL/配置 | — | 编译通过；契约冻结 |
| **T1** | 数据层/去重/归档 | `database/news/*`、`handler/news/store/*` | T0 | `NewsStore` | DTO | 单测+编译 |
| **T2** | 采集 | `handler/news/fetch/*` | T0 | `NewsFetcher` | DTO、`NewsConfig` | 单测+编译 |
| **T3** | AI 整理 | `handler/news/curate/*` | T0 | `NewsAiCurator` | DTO、LLM Provider | 单测+编译 |
| **T4** | 页面生成+交互 | `handler/news/render/*` | T0 | `NewsPageBuilder` | `DailyReport`、`ReportMeta` | golden 单测+编译 |
| **T5** | 托管+访问 | `handler/news/hosting/*`、`controller/NewsViewController.java` | T0 | `NewsHosting`、HTTP `/news/**` | HTML 字符串 | 单测(MockMvc)+编译 |
| **T6** | 调度编排 | `schedule/DailyNewsSchedule.java` | T0（接口） | 定时任务 | 五大接口 | 编排单测+编译 |
| **T7** | 部署(RSSHub) | `bb-bot.yaml`(追加段)、`application-prod.yml`(news覆盖) | 无 | RSSHub 服务 | — | dry-run 通过 |

**不重合保证**：
1. 每个任务的"独占文件/包"两两不相交；唯一被多任务读取的是 T0 的契约（只读）与 `application.yml`（仅 T0 写 `news` 段）。
2. 去重职责按 §5 切成 L1（T1）/L2（T3），层次不同不重合。
3. 交互三件套（筛选/搜索/跨天导航）**全部归 T4**；T5 只提供"可用日期列表"作为数据，不碰前端。
4. T6 只编排、不实现任何业务逻辑；业务逻辑全在 T1–T5。

## 8. 交互设计细节（归 T4）

- **分类筛选**：顶部 sticky tab，点击 `data-cat` 过滤 `.card`，更新计数（已在 demo 验证）。
- **折叠**：卡片内 `<details>` 展开原文要点。
- **站内搜索**：tab 行旁一个输入框，`input` 事件按关键词（标题+摘要，小写）过滤；与分类筛选**叠加生效**（两者交集）。
- **跨天往期导航**：
  - 单日页顶部：`← 前一天 / 选择日期 ▾ / 后一天 →`，选项来自 `availableDates`（无数据的日期禁用）。
  - 归档索引页 `/news/archive`：按日期倒序列出近 30 天，每项显示日期 + 条数 + 入口链接。
- 纯静态实现：单页所需的"可用日期"在生成时注入为内联 JS 数组，无需后端接口。

## 9. 测试策略（每任务"完成"的硬性部分）

- 每个代码任务（T1–T6）交付物**必须含单测**且**本地 `mvn` 编译+测试通过**（对齐仓库测试习惯：解析类用 fixture、渲染类用 golden 快照、编排类用 mock）。
- 集成验证（非单个任务范畴，单列后续步骤）：全链路在本地用真实源跑一次，产出真实 HTML 并人工核对，再接 auto-deploy。
- 本地编译命令遵循仓库约定（mvn 绝对路径 + JAVA_HOME 17 + 本地仓库参数）。

## 10. 风险与对策

| 风险 | 对策 |
|---|---|
| 契约后续变更牵动所有任务 | T0 先评审冻结；变更走集中修订并通知 |
| RSSHub 路由失效/反爬 | 单源隔离（T2）；源可配可替换；调研已记录淘汰清单 |
| LLM 输出不合规 JSON | T3 严格解析 + 降级；分类越界纠正 |
| HTML 过长 | 每类条数上限 `perCategoryLimit`（T4 遵守） |
| auto-deploy 被 YAML 破坏 | T7 dry-run + 保留镜像占位行（遵 CLAUDE.md 硬规则） |

## 11. 落地顺序建议

1. **先合 T0**（契约冻结）。
2. **并行**展开 T1–T6 + T7。
3. 各任务单测+编译通过后合入。
4. 集成联调（全链路真实源跑一次 + 人工核对 HTML）。
